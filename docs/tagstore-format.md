# Tag Store Format (version 1)

Reference specification for the on-disk tag store that archnav reads and
writes (GUI tags column, Ctrl+T editor, `t:` search filter, and the
`archnav tag` CLI). The format is designed for filesystems without extended
attribute support - CIFS/SMB mounts of a NAS being the motivating case -
where the native Linux tagging convention (the `user.xdg.tags` xattr used
by KDE Dolphin and Baloo) fails outright.

The format originated in a standalone Python tool (tagdex); archnav's Rust
implementation is byte-compatible and is now the reference. Compatibility is
pinned by tests against fixtures generated by the original implementation.

## Layout

A store covers one directory tree. Its root contains:

```
<root>/.tagstore/index.json   # the entire tag database
<root>/.tagstore/lock/        # transient; exists only while a writer holds the lock
```

## index.json

```json
{
  "entries": {
    "Top Picks/menu.pdf": {
      "fp": "c374106b0e1eef9844fc247e5a222d47",
      "mtime_ns": 1784520622186633212,
      "size": 12,
      "tags": ["menu", "restaurant"]
    }
  },
  "version": 1
}
```

- Keys of `entries` are slash-separated paths relative to the store root.
  Never absolute: the tree can be remounted elsewhere or copied wholesale
  and the tags follow.
- `tags` is a sorted list of unique strings. Tags may contain spaces,
  colons, and any unicode; commas and control characters are invalid
  (the comma is the separator in the mirrored xattr).
- `size` is the file size in bytes; `mtime_ns` is st_mtime in integer
  nanoseconds. Together they are the cheap staleness check: if both match
  the file's current stat, the stored fingerprint is trusted without
  re-reading the file.
- `fp` is the content fingerprint (below).
- Entries with no tags are removed, never stored empty.
- Serialization is deterministic - keys sorted, 2-space indent, unescaped
  UTF-8, trailing newline - so the file diffs cleanly under version control
  and sync tools.
- `version` is checked by readers before anything else; a reader must
  refuse a version greater than it knows.

## Fingerprint

blake2b with a 16-byte digest, fed with:

1. the file size as 8 bytes little-endian,
2. the whole content when size <= 131072 bytes (2 x 64 KiB),
   otherwise the first 64 KiB followed by the last 64 KiB.

Rendered as 32 lowercase hex characters.

Reading at most 128 KiB keeps fingerprinting fast on high-latency network
mounts regardless of file size. Deliberate tradeoff: two same-size files
differing only strictly between the head and tail chunks share a
fingerprint. Fingerprints locate files for tag relinking (`repair`); they
are not integrity checksums.

## Write protocol

1. Acquire the store lock (below).
2. Read index.json, apply mutations in memory.
3. Write to a temporary file in the same directory
   (`index.json.tmp.<pid>`), fsync, then rename over index.json.
   Same-directory rename is atomic on local filesystems and on the SMB
   server for CIFS mounts, so readers never observe a partial index.
4. Release the lock.

Readers take no lock; the atomic rename makes lock-free reads safe.

## Lock protocol

The lock is the directory `.tagstore/lock/`, created with mkdir - atomic on
local filesystems and over SMB, unlike O_EXCL opens or byte-range locks
(which is also why the index is JSON rather than SQLite: SQLite's POSIX
locking is unreliable over CIFS). The holder writes `lock/owner.json`
(host, pid, timestamp) for diagnostics only.

Staleness: a lock whose directory mtime is older than 600 seconds may be
broken by a waiting writer (with a warning). The mtime is assigned by the
filesystem server, so the judgment is consistent across client machines
with skewed clocks. Writers poll every 100 ms and give up after 10 seconds
with an error naming the current holder.

## xattr mirror

After a successful write, the writer mirrors each file's final tag list to
the `user.xdg.tags` extended attribute (comma-separated), so KDE Dolphin
and Baloo pick tags up natively on filesystems that support user xattrs.
Failures (ENOTSUP on CIFS, permissions) are silently ignored, and the
xattr is never read back: the index is the sole source of truth.

## Repair semantics

`repair` reconciles the index with the tree:

- An entry whose path no longer exists is matched against untracked files
  by size, then fingerprint. Exactly one match: the entry is relinked to
  the new path, keeping its tags. Multiple matches (identical duplicates):
  reported as ambiguous, never guessed. No match: reported missing
  (dropped only with `--prune`).
- An entry whose file changed in place (size or mtime_ns differ) gets its
  fingerprint recomputed, keeping its tags.

A file that is both renamed and modified before repair runs cannot be
relinked (the fingerprint no longer matches) and is reported missing.
